# Публикация с помощью командной строки

Публикацию можно запустить через командную строку с помощью инструмента Directum Launcher, вызвав утилиту DeploymentToolCore .

Важно. Перед первой публикацией убедитесь, что она настроена. Подробнее см. в руководстве администратора, раздел «Утилита DeploymentToolCore» ( Linux , Windows ).

Для публикации решения в систему с существующими пакетами требуется указать полный комплект: базовые прикладные, установленные и новые пакеты.

Также вы можете выполнить публикацию с помощью DeploymentToolCore без использования Directum Launcher. Для этого запустите утилиту через Docker .

При публикации можно использовать основные и дополнительные команды утилиты, которые расширяют ее возможности. При этом некоторые выполняемые действия можно запустить разными командами.

При возникновении ошибок определите причину по коду возврата .

## Публикация

- 1. Запустите командную строку от имени администратора и перейдите в папку с Directum Launcher.
- ВАЖНО. Утилита DeploymentToolCore добавляется вместе с компонентом платформы системы. Поэтому утилиту необходимо запускать из папки с инструментом Directum Launcher, с помощью которого установили систему. Иначе возникнет ошибка.
- 2. Опубликуйте разработку. Для этого выполните команду:

./do.sh dt deploy --package="<Путь к пакету 1>;<Путь к пакету 2>;<Путь к пакету N>"

do dt deploy --package="<Путь к пакету 1>;<Путь к пакету 2>;<Путь к пакету N>"

- В параметре --package укажите публикуемые пакеты через точку с запятой.
- Пример команды:

./do.sh dt deploy --package="/srv/CustomDev/Dev1.dat;/srv/CustomDev/Dev2.dat"

do dt deploy --package="D:\CustomDev\Dev1.dat;D:\CustomDev\Dev2.dat"

- ПРИМЕЧАНИЕ. Если в базе данных уже есть публикуемая версия прикладной разработки, то повторно она не опубликуется. Чтобы это сделать принудительно, выполните команду с параметром --force .

При выполнении команды публикуется разработка, инициализируются модули и применяются настройки бизнес-процессов и интерфейса по умолчанию.

СОВЕТ. При публикации прикладной разработки система недоступна. Если нужно указать несколько пакетов разработки, то затрачивается время на их объединение, что может дополнительно увеличить период простоя. Чтобы ускорить обновление, с помощью команды dt merge_packages заранее подготовьте объединенный пакет с нужными решениями, а затем при проведении плановых работ опубликуйте его.

При публикации используются имя пользователя и пароль, указанные в конфигурационном файле config.yml. Если публикацию необходимо запустить от имени другого пользователя, укажите параметры --user и --password :

./do.sh deploy --package="<Путь к пакету 1>;<Путь к пакету N>" --user="<Имя пользователя>" --password="<Пароль пользователя>"

do dt deploy --package="<Путь к пакету 1>;<Путь к пакету N>" --user="<Имя пользователя>" --password="<Пароль пользователя>"

ВАЖНО. Логины и пароли необходимо записывать в двойных кавычках, если они содержат спецсимволы или пробелы, например, логин "Service User" и пароль "~!@55555" . Если кавычки уже используются в пароле, их необходимо экранировать. Например, пароль вида pas~!@"88 нужно записывать как "pas~!@""88" .

Основные и дополнительные команды утилиты расширяют ее возможности. При этом некоторые выполняемые действия можно запустить разными командами.

## Основные команды утилиты

В таблице приведены основные команды утилиты. В операционных системах Microsoft Windows и Linux формат команд одинаковый, отличается только способ их вызова:

./do.sh <команда>

do <команда>

Команда | Описание
dt export-package --export_package <dev_package> --configuration <configuration> --root <path> [--repositories <paths>] | Создать пакет разработки
dt merge_packages "<Путь к объединенному пакету>" --packages="<Пути к пакетам>" | Объединить пакеты разработки без публикации. В параметре --packages через точку с запятой укажите пути к пакетам и их названия. Пример команды: ./do.sh dt merge_packages "/app/union.dat" --packages="/app/first.dat;/app/second.dat"
dt merge_packages "<Путь к объединенному пакету>" -- package_from_component ="<Названия компонентов>" | Объединить пакеты разработки без публикации. В параметре -- package_from_component укажите названия компонентов с нужными пакетами. Пример команды: ./do.sh dt merge_packages "/app" --package_from_component="base;memo;agile;genai"
dt increment_version --root <path> --repositories <paths> | Увеличить номер версии модулей и решений
dt set_version --version <version> --root <path> --repositories <paths> | Задать номер версии модулей и решений
dt deploy --package="<Путь к пакету>" | Запустить публикацию и инициализацию разработки, применить настройки бизнес-процессов по умолчанию. Если пакет уже опубликован, то он повторно не публикуется
dt deploy --package="<Путь к пакету>" --force | Запустить публикацию принудительно, независимо от того, установлено решение или нет. Например, если прикладная разработка уже есть и для устранения инцидентов ее нужно перепубликовать. При этом также выполняется инициализация разработки и применяются настройки бизнес-процессов по умолчанию
dt deploy --package="<Путь к пакету>" --init | Запустить публикацию и инициализацию разработки. Если нужно выполнить публикацию без инициализации и применения настроек, например при обновлении прикладной разработки в рамках версии, то в ключе --init укажите значение False : dt deploy --package="<Путь к пакету>" --init=False
dt deploy --package="<Путь к пакету>" --settings | Запустить публикацию разработки и применить настройки бизнес-процессов и интерфейса по умолчанию
dt deploy --package="<Путь к пакету>" --dev | Запустить публикацию разработки
dt deploy --package="<Путь к пакету>" --distributed | Запустить публикацию в распределенную систему
dt deploy --init | Запустить инициализацию разработки.
dt deploy --settings | Применить настройки бизнес-процессов по умолчанию. Настройки могут содержать ссылки на сущности, которые создаются при инициализации, поэтому параметр выполняется после параметра --init
dt get_deployed_solutions | Получить список опубликованных решений
dt get_applied_solutions_info --package_path="<Путь к пакету>" | Получить информацию о решениях из пакета разработки
dt init_and_apply_settings | Запустить инициализацию разработки и применить настройки бизнес-процессов и интерфейса по умолчанию
dt init | Запустить инициализацию
dt remove_solutions --solution_names="<Имена удаляемых решений>" | Удалить решения. В параметре --solution_names перечислите имена удаляемых решений через пробел, например: dt remove_solutions --solution_names="<Solution1 Solution2>" Если удалить решение необходимо от имени другого пользователя, укажите параметры --user и --password

## Дополнительные команды утилиты

C помощью ключа --command можно указать дополнительные параметры запуска. В операционных системах Microsoft Windows и Linux формат команд одинаковый, отличается только способ их вызова:

./do.sh dt run --command="<Параметры>"

do dt run --command="<Параметры>"

Важно. Если логины и пароли содержат спецсимволы и пробелы, то перед двойными кавычками нужно добавить экранирующий символ \ . Например, логин \"Service User\" и пароль \"~!@55555\" . Если в пароле используются кавычки, то их необходимо также экранировать с помощью символа " , то есть указать две двойные кавычки. Например, пароль вида 1Qwerty" нужно записать как \"1Qwerty""\" . Пример команды:

do dt run --command="-n \"Service User\" -p \"1Qwerty""\""

Параметр | Описание
-n (или --name) -p (или --password) | Имя и пароль пользователя для публикации. Пример: do dt run --command="-n username -p password -x"
-d (или --development-package) | Путь к публикуемому пакету разработки
-m (или --merge-only) | Путь для сохранения объединенного пакета разработки. При выполнении команды публикация не запускается. Пример: do dt run --command="-d D:\CustomDev\first.dat D:\CustomDev\second.dat -m D:\CustomDev\union.dat" Если название объединенного пакета не задано, а указана только папка, то имя пакета генерируется по шаблону: MergedPackage-<yyyy-MM-dd_HH-mm-ss>.dat
-x (или --initialize) | Запустить инициализацию. По умолчанию инициализируются модули всех решений. Если инициализацию нужно выполнить только для определенных решений, через пробел укажите их названия. Можно указывать или оба параметра -d и -x , или один из них
-s (или --apply-settings) | Применить настройки бизнес-процессов и интерфейса по умолчанию. Настройки могут содержать ссылки на сущности, которые создаются при инициализации, поэтому параметр -s выполняется после -x
--settings-localization="<Путь к XLSX-файлу со строками>" | Импортировать строки локализации настроек
-l (или --list-deployed-solutions) | Отобразить список всех опубликованных модулей
-r (или --remove-solutions) | Имена решений, которые необходимо удалить. Указываются через пробел
-y (или --settings) | Путь до конфигурационного файла утилиты
--distributed | Публикация в распределенную систему . Пример: do dt run --command="-n Administrator -p 11111 -d "D:\CustomDev\DevPackage.dat" --distributed" Параметр --distributed нельзя использовать совместно с ключами -x и -r
--help | Показать список и описание всех параметров утилиты. Пример: do dt run --command="--help"

## Коды возврата ошибок

При возникновении ошибки выводится код возврата, который позволяет определить ее причину:

Код возврата | Описание
0 | Публикация завершилась без ошибок
1 | Ошибка возникла до отправки пакета разработки на сервер, например, во время: • проверки аргументов командной строки; • запроса метаданных с сервера; • чтения пакета разработки; • формирования пакета развертывания
2 | Ошибка возникла при передаче пакета разработки на сервер или при публикации, например: • сетевая ошибка при отправке пакета развертывания на сервер; • ошибка публикации пакета, когда сервер недоступен; • ошибка публикации пакета, когда версия утилиты DeploymentToolCore отличается от версии веб-сервера. Публикация запускается, если версии совпадают на уровне редакции релиза (Directum RX 26.1.0.XXXX)
3 | Ошибка возникла во время инициализации модулей при: • отправке запроса на инициализацию; • выполнении инициализации модулей на веб-сервере
4 | Ошибка возникла при применении настроек по умолчанию из пакета разработки
5 | Ошибка возникла при импорте настроек
6 | Ошибка возникла при экспорте настроек
7 | Ошибка возникла при экспорте пакета разработки
8 | Ошибка возникла при изменении номера версии модулей и решений
