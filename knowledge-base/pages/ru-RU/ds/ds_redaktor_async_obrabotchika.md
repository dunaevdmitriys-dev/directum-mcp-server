# Редактор асинхронного обработчика

Чтобы задать или изменить настройки асинхронного обработчика, в редакторе модуля на вкладке «Асинхронные обработчики» нажмите на его название в списке асинхронных обработчиков. Открывается редактор, в котором:

- задаются именование , параметры и свойства асинхронного обработчика;
- разрабатывается обработчик события «Выполнение», к коду которого доступен переход из среды разработки.

## Именование

*Имя асинхронного обработчика. Значение должно быть уникальным в рамках одного модуля, может состоять из букв латинского алфавита и цифр, а также начинаться с буквы. *Описание асинхронного обработчика. Для локализации описания перейдите к полю и нажмите на кнопку . *Идентификатор асинхронного обработчика. Поле заполняется автоматически и используется при настройке конфигурационного файла в секции сервиса асинхронных событий .

Параметры

Имя параметра. Значение должно быть уникальным в рамках одного асинхронного обработчика, может состоять из букв латинского алфавита и цифр, а также начинаться с буквы. Тип данных параметра. Возможные значения: Строка , Целое , Дата , Логическое , Идентификатор , Вещественное . ПРИМЕЧАНИЕ. Параметры асинхронных обработчиков ограничены простыми типами. Например, нельзя использовать тип Бинарные данные , так как передача файлов через массив байт приведет к большому потреблению ресурсов на сервисе Worker . С файлами надо работать только через потоки.

Свойства

Повторить . Повторное выполнение асинхронного обработчика на сервисе Worker при возникновении ошибки. Возможные значения: • С увеличением интервала до 1 часа, затем до 2 . В течение первого часа обработчик выполняется с увеличивающимся по экспоненте интервалом между попытками повтора. После этого они выполняются каждый час. Через 12 часов интервалы между повторами увеличиваются до 2 часов. По умолчанию асинхронный обработчик прекращает выполнение после 50 попыток; • Через равные промежутки времени . Интервал повтора в минутах указывается в свойстве *Период (в минутах). По умолчанию асинхронный обработчик повторно выполняется каждые 15 минут и прекращает выполнение после 100 попыток. Максимальное количество повторов при возникновении ошибки. По умолчанию 50 .

Обработчики

```csharp
public virtual void GrantAccessRightsToDocument(Sungero.Docflow.AsyncHandlers.GrantAccessRightsToDocumentInvokeArgs args)
{
long documentId = args.DocumentId;
long ruleId = args.RuleId;

Logger.DebugFormat("TryGrantRightsByRule: start grant rights for document {0}, rule {1}", documentId, ruleId);

var isGranted = Docflow.Functions.Module.GrantRightsToDocument(documentId, ruleId);

if (!isGranted)
{
Logger.DebugFormat("TryGrantRightsByRule: cannot grant rights for document {0}, rule {1}", documentId, ruleId);
args.Retry = true;
}
else
{
Logger.DebugFormat("TryGrantRightsByRule: success grant rights for document {0}, rule {1}", documentId, ruleId);
}
}
```

Аргументы обработчика событий «Выполнение»

В событии асинхронного обработчика можно использовать аргументы: Аргумент Тип Описание args.MaxRetryCount int Максимальное количество повторов обработчика. Если лимит повторов превышен, то асинхронный обработчик не выполняется args.NextRetryTime Nullable<DateTime> Дата и время следующего повтора обработчика args.Retry bool Признак, который определяет, нужно ли повторно выполнять асинхронный обработчик после возникновения ошибки args.RetryIteration int Счетчик повторов. С помощью параметра можно понять, какая попытка повтора сейчас выполняется args.RetryReason string Произвольная информация, которая записывается в лог-файлы. Например, причина повтора обработчика, результат его выполнения СОВЕТ. Для логирования используйте аргумент RetryReason вместо метода DebugFormat() . Информация из этого аргумента фиксируется в лог-файле и ее можно использовать для анализа работы обработчиков с помощью решения «Мониторинг Directum RX»: Для логгера AsyncHandlerOperation в атрибуте span записываются значения: • retry – признак того, что выполнение обработчика отложено; • maxIteration – максимальное количество итераций; • systemRetry – признак того, что система отправила обработчик на переповтор; • retryReason – информация, которая записывается при необработанном исключении, или сообщение, заданное разработчиком в аргументе RetryReason . ПРИМЕЧАНИЕ. Если при выполнении обработчика возникает необработанное исключение, то он отправляется на переповтор. В поле systemRetry записывается значение true , а в поле retryReason записывается класс и текст ошибки.
