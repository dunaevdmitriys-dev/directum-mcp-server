# Unstuck — Помощник мышления

> Ты не застрял. Ты просто ещё не нашёл правильный подход. Этот skill помогает искать.

## Когда активировать

- Задача нестандартная, нет готового рецепта
- Fix не сработал с первого раза
- Непонятно как реализовать фичу на платформе
- Поведение системы противоречит ожиданиям

## Как думать: 4 источника вдохновения

### 1. Найди рабочий пример в платформе
Платформа Directum RX — это сотни сущностей, написанных профессионалами. Почти любая задача уже решена где-то в base/.

```
MCP: search_metadata name=<ключевое_слово>
MCP: search_metadata type=<тип_свойства>
MCP: extract_entity_schema entity=<похожая_сущность>

Примеры поиска:
- Нужен справочник с деревом? → search_metadata "IsHierarchical"
- Нужен документ с согласованием? → search_metadata "ApprovalTask"
- Нужна связь многие-ко-многим? → search_metadata "CollectionPropertyMetadata"
- Нужно действие на карточке? → search_metadata "ActionMetadata"
- Нужен виджет на обложке? → search_metadata "WidgetMetadata"
```

Платформенный код — лучшая документация. Если платформа делает X, значит X возможен.

### 2. Посмотри как устроены реальные решения
```
Репозиторий решения:
- Платформенные модули (эталон) — внутри контейнеров RX, доступны через MCP: search_metadata / extract_entity_schema
- {project_path}/source/ — модули текущего проекта (реальный опыт)

Каждый платформенный модуль — это production-quality код с:
- Правильной структурой .mtd
- Рабочими workflow
- Обработчиками событий
- Примерами интеграций
```

### 3. Подумай об альтернативных архитектурах
```
Прямой путь не работает? Вот альтернативы:

Вместо сложной DDS-сущности:
  → Remote Component (React-контрол внутри RX карточки)
  → WebAPI endpoint + клиентская функция
  → CrmApiV3 + отдельная таблица PostgreSQL

Вместо сложного workflow:
  → Простой workflow + AsyncHandler для логики
  → Задача без workflow + серверная функция с ручным управлением
  → Два простых workflow вместо одного сложного

Вместо кастомной формы:
  → ControlGroup с умной логикой Showing/Refresh
  → Remote Component для нестандартного UI
  → InputDialog для одноразового ввода

Вместо модификации платформенной сущности:
  → Перекрытие (override-entity) с добавлением свойств
  → Связанная кастомная сущность с NavigationProperty
  → Отдельный справочник-настройка
```

### 4. Спроси пользователя
```
НЕ крути одно и то же. После 2-х неудачных попыток — СПРОСИ:

Хороший вопрос:
"Я попробовал [подход A] и [подход B], оба не сработали потому что [причина].
 Вижу два варианта: [вариант 1] или [вариант 2].
 Какой ближе к тому что нужно? Или может есть идея которую я не рассматривал?"

Плохой вопрос:
"Не получается. Что делать?" ← нет контекста, нет вариантов
```

## Шаблоны мышления для типовых тупиков

### "Не знаю как это сделать на платформе"
```
1. Сформулируй задачу одним предложением
2. MCP: search_metadata — ищи ключевые слова в платформе
3. Нашёл похожее? → extract_entity_schema → изучи как сделано
4. Не нашёл? → Это редкая задача. Подумай об альтернативной архитектуре (п.3 выше)
5. Предложи пользователю 2-3 варианта с плюсами/минусами
```

### "Ошибка непонятная, в интернете ничего"
```
1. MCP: trace_errors — полный контекст ошибки
2. Ищи НЕ текст ошибки, а КОНТЕКСТ: какой файл, какая операция, какой этап
3. MCP: analyze_solution action=health — может проблема в окружении?
4. Сравни с рабочим примером из платформы (MCP: extract_entity_schema) — в чём разница?
5. Попробуй минимальный пример: одна сущность, одно свойство. Работает? Добавляй по элементу.
```

### "Работало, перестало"
```
1. Что менялось? git log, diff, последние действия
2. DDS кэширует .mtd → перезапусти DDS
3. IIS/сервисы кэшируют DLL → перезапусти
4. Satellite DLL устарела → проверь AppliedModules/ru/
5. Стенд обновился → MCP: analyze_solution, проверь версии
```

### "Задача слишком большая, не знаю с чего начать"
```
1. Разбей на этапы по 1-2 часа каждый
2. Каждый этап должен быть проверяемым (можно импортировать и увидеть результат)
3. Начни с самого рискованного (непонятного) — если он не решаем, лучше узнать сразу
4. /pipeline fast — для быстрого старта без тяжёлого проектирования
```

## Главное

**Claude — не оракул. Claude — напарник.** Если напарник 2 раза попробовал и не получилось — он спрашивает коллегу, а не повторяет в третий раз. Спрашивай пользователя. Предлагай варианты. Думай вслух.
